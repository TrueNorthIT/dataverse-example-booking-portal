# CLAUDE.md

Guidance for AI assistants working in this repository.

## What this is

A citizen booking portal ("Leeds City Council — Book a Service") — a React 19 + TypeScript SPA that talks **only** to the [Dataverse Contact API](https://github.com/TrueNorthIT/dataverse-contact-api) (`citizenbooking` scope). The API handles Microsoft Entra External ID auth, RBAC, and proxies to Microsoft Dataverse; the SPA never calls Microsoft APIs directly. Citizens sign in via Entra External ID (MSAL PKCE, `@azure/msal-browser` + `@azure/msal-react`); the app's `useAuth()` adapter (`src/auth/useAuth.ts`) wraps MSAL. See `README.md` for the full per-page API walkthrough and `OVERVIEW.md` for the architecture summary.

## Commands

```bash
npm run dev            # Vite dev server (http://localhost:5173)
npm run build          # tsc -b (typecheck, incl. tests) + vite build
npm run lint           # ESLint (flat config)
npm test               # Vitest — hermetic unit/hook/component suite
npm run test:coverage  # Vitest with the coverage gate
npm run test:e2e       # Playwright (needs a running app + Entra External ID)
npm run test:live      # OPT-IN: hits real Dataverse via .env (never in CI/coverage)
npm run generate:types # Regenerate src/types/generated.ts from the API schema
npm run schema|seed|clean   # Provision / seed / tear down demo Dataverse data
```

Before committing, run `npm run lint && npm run build && npm run test:coverage` — all must pass.

## Architecture & layout

- **`src/lib/`** — pure, framework-free business logic and the source of truth for booking rules. Keep logic here (it's directly unit-tested) and keep React components thin. Key modules: `availability.ts`, `slotGridLayout.ts`, `slotCell.ts`, `busyness.ts`, `venue.ts`, `bookings.ts`, `pricing.ts`, plus `constants.ts`, `logger.ts`, `queryKeys.ts`.
- **`src/hooks/`** — thin TanStack Query wrappers over the SDK. `useApi.ts` holds the generic list/get/mutation helpers; `useDataverse.ts` builds the SDK clients (the public singleton + the authenticated `useDataverse()`).
- **`src/components/`** — `citizen/` and `common/` are app components; **`ui/` is vendored shadcn/ui — don't hand-edit it.**
- **`src/contexts/`** — `DemoContext`/`ApiStatsContext` (plain), and `RealtimeContext`/`PresenceContext` (SignalR + Azure Web PubSub).
- **`src/types/generated.ts`** — generated by `npm run generate:types`. **Never edit by hand.**

### API tiers
`public/*` (no auth, read-only), `me/*` (caller's own contact, row-scoped), `all/*` (unscoped, needs `:all` permission). The booking create is a single deep-insert `POST me/servicebooking` with the venue booking nested under `tn_booking`; cancel patches the citizen record on `me` then the venue booking on `all`.

## Conventions

- Put business logic in `src/lib/` as pure functions and test it directly — don't bury it in components or re-implement it in tests.
- No magic numbers/strings — add them to `src/lib/constants.ts` (e.g. `BUSYNESS`, `PRESENCE`, `BOOKING_STATUS_NAME`).
- No raw `console.*` — use `src/lib/logger.ts` (`debug`/`info` are silenced outside dev).
- Use `queryKeys` (`src/lib/queryKeys.ts`) for every React Query key.
- TypeScript is strict; `noUnusedLocals`/`noUnusedParameters` are on. No `any` in `src/`.

## Testing

- Vitest + React Testing Library (jsdom). The `useAuth()` adapter is mocked via `tests/setup/auth-mock.ts` (so no real `@azure/msal-*` imports run); the Dataverse client and Web PubSub are mocked in `__mocks__/` and all are activated in `tests/setup/vitest.setup.ts`.
- Use the shared helpers in `tests/setup/test-utils.tsx`: `renderWithProviders`, `createHookWrapper`, `mockClient` (drive responses via `mockClient.public.list.mockResolvedValueOnce(...)`), and `setAuthState` (aliased `setAuth0State` for back-compat).
- Coverage gate: 90% lines/statements/functions, 85% branches. Excluded: `components/ui/**`, generated types, `main.tsx`/`App.tsx`, the realtime contexts (covered by E2E) and `useDataverse.ts`.
- `tests/live/` is the opt-in real-Dataverse suite — keep it out of the default run.

## Demo-only shortcuts (don't mistake for production patterns)

- `src/lib/pricing.ts` and `src/lib/locations.ts` hold hardcoded venue rates/capacity and persona/venue locations, keyed by name — pending API columns.
- `PresenceContext` mints the Web PubSub client-access token **in the browser**. Demo only — issue it server-side in production.
- Stripe runs in test mode and is confirmed client-side (no webhook).
